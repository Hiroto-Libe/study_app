ベテランエンジニアとして、ご提示いただいた「学習アプリ 詳細設計書 v1.0」を「レビュー依頼プロンプト」の観点に基づき、優先度（【高】【中】【低】）を付与してレビューいたしました。

全体として、ディレクトリ構成からSQL、TypeScriptの型定義まで非常に精緻に組まれており、Claude Code等のAIエージェントに渡しても高い精度で実装が進むことが期待できる優れた設計書です。

---

### 📝 詳細設計書レビュー報告書

#### 1. TypeScript 型設計（2章）
* **【高】QuizSessionState の遷移漏れ**: 音声入力や書き取り判定において、APIとの通信中を示す `loading` 状態や、通信エラー時の `error` 状態がState定義に含まれていません。これがないと、ローディング中にボタンを連打される等のバグに繋がります。
* **【中】any の排除**: 一部のAPIレスポンス受け取り箇所で `any` の使用が想定されます。`zod` 等を用いたスキーマバリデーションを導入し、`z.infer<T>` で型を導出する設計を明記すると、実行時の型安全性が劇的に向上します。

#### 2. lib/ モジュール設計（3章）
* **【高】claude.ts のリトライ設計**: ネットワーク不安定時やClaude APIの過負荷時に備え、単純なリトライではなく、指数バックオフ（Exponential Backoff）を用いたリトライ処理の組み込みを推奨します。
* **【中】supabase.ts のSSR対応**: Next.js 14のApp Routerを使用しているため、`createClient` を呼び出す際に `cookies` ヘッダーを適切に処理する `createBrowserClient` / `createServerClient` の使い分け（`@supabase/ssr`）が設計書内でより強調されるべきです。

#### 3. API Routes 設計（4章）
* **【高】POST /api/study/answer のトランザクション**: 経験値（XP）の加算、ストリークの更新、苦手項目の記録は、どれか一つが失敗しても不整合が起きます。Supabase（PostgreSQL）の `RPC` を使用してデータベース側で一括更新するか、トランザクション的な一貫性を担保する設計が必要です。
* **【低】悪用リスク**: Phase 1では認証なしとのことですが、APIエンドポイント（URL）が判明すると外部からXPを不正に加算されるリスクがあります。短期的には Vercel の環境変数で簡易な API Key を設定するだけでも効果があります。

#### 4. コンポーネント設計（5章）
* **【中】StrokeAnimation.tsx (KanjiVG)**: SVGの描画において、Reactの再レンダリングによってアニメーションが途切れる可能性があります。`useEffect` 内で直接DOM（SVG要素）を操作するか、`framer-motion` 等のライブラリとの親和性を考慮した設計を追記すると実装者が迷いません。
* **【低】NumberKeypad.tsx**: 小4算数では小数点やマイナスの概念が登場するため、それらのキーの表示・非表示を `props` で切り替えられる柔軟性を持たせておくと、将来的な拡張が容易になります。

#### 5. エラーハンドリング・環境構築（6〜9章）
* **【中】ErrorBoundary の配置**: `learn/layout.tsx` だけでなく、クイズ回答中の予期せぬエラーで学習意欲を削がないよう、各クイズコンポーネント単位で「やり直しボタン」付きの ErrorBoundary を配置することを推奨します。
* **【低】セットアップ手順**: `KanjiVG` のSVGファイルを `public/kanjivg/` に配置する際、ファイル名（Unicode）と漢字の対応付けを自動化するスクリプトの存在を明記しておくと、手動作業によるミスを防げます。
